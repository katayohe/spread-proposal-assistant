# 4枚目（API・計算資源費用 算定根拠）ガイド — AWS 前提版

本スキルは研究者が **AWS** を計算資源として使う前提。価格は **AWS Price List Bulk API（公開・認証不要）** から都度取得する。ヘルパーは `scripts/fetch_aws_price.py`。

## ⚠️ 研究期間の厳守（算定月数の上限）

第2回公募の研究期間は **交付決定日〜令和9年2月5日（最大約6ヶ月）** である。算定根拠で使用する月数（`months`）は **この研究期間を超えてはならない**。

- **禁止例**：`× 12ヶ月`、`/年`、`年額○円` — 研究期間を超えた計算は制度上の矛盾であり、審査時に不備と判断される
- **正しい例**：`× 6ヶ月`（研究期間全体）、`× 3ヶ月`（後半のみ使用）など、研究期間内に収まる月数を使う
- 交付決定日が未確定の段階では **最大6ヶ月** を上限とし、算定根拠にその旨を明記する
- 「年額」「年間」ベースの料金表示を見てもそのまま使わず、必ず研究期間内の月数に換算して計算すること

## ⚠️ 重大な原則：勝手にサードパーティ API を提案しない

過去の事故で、別エージェントが研究内容に合わせて **Elsevier Text Mining API（論文全文取得）** や **Qdrant Cloud（ベクトルDB）** などのサードパーティ API を**ユーザーに確認せず勝手に**4枚目の API 費用テーブルに埋めた事例があった。

本スキルでは以下を厳守する：

1. **API 費用テーブル（行9〜18）に記載できるのは AWS サービスのみ**（Bedrock, EC2 推論エンドポイント, Comprehend, Translate, Polly, Rekognition, Textract 等）。
2. **AWS 以外のサードパーティ API**（Elsevier, Qdrant, OpenAI 直接, Anthropic 直接, Pinecone, Weaviate, Cohere, Hugging Face Inference Endpoints 等）は、**ユーザーが自発的に「使いたい」と明示した場合のみ**記載する。エージェント側から提案・推測しない。
3. ヒアリングしていないサービスを「研究内容に合いそうだから」という理由で**創作・追加することは厳禁**。SPReAD は AWS 利用が前提の制度設計であり、サードパーティ API を計上すると審査時に不要な疑義を招く。
4. 実装ヘルパー `compute_aws_cost.py` および `fetch_aws_price.py` は AWS Price List Bulk API のみをデータソースとする。Elsevier 等のサードパーティ料金は本スキルでは取得手段を持たない。
5. もし研究内容上どうしてもサードパーティ API が必要そうな場合は、エージェントが勝手に埋めるのではなく、**ユーザーに「この研究で○○を使う予定はありますか？」と AskUserQuestion で確認**してから記入する。

**単位ルール**：
- ヒアリング・ドラフト・メモは **万円** で統一
- xlsx 様式（4枚目の「金額(千円)」欄）への書き込み時のみ **千円** に換算（`千円 = 万円 × 10`）
- 千円未満切り捨て（様式の注記に従う）

## AWS 費用算定プロトコル

### Step 1. 用途の整理

ユーザーから以下を引き出す：

1. **用途分類**：学習（training）／推論（inference）／データ前処理／埋め込み生成／ベクター検索 等
2. **モデル規模**：〜1B, 1〜10B, 10〜70B, 70B+ パラメータ
3. **データ量**：学習データ全体サイズ（GB/TB）、想定エポック数
4. **稼働想定**：連続稼働（例：1週間×8h）か、断続稼働か、研究期間全体での総時間
5. **マネージドサービス利用**：SageMaker、Bedrock（Claude/Titan等）、EC2 生インスタンス

### Step 2. AWS 公開料金 API で価格取得

**MCP は使わない**。アクセスフローは以下：

1. `https://pricing.us-east-1.amazonaws.com/offers/v1.0/aws/index.json`（~100KB）を取得
2. 対象サービスの `currentRegionIndexUrl` → 対象リージョンの `currentVersionUrl` を解決
3. regional 本体 JSON は**ストリーム読み**（`ijson`）、対象 SKU で早期終了

`scripts/fetch_aws_price.py` がこれを行う。典型呼び出し：

```bash
python3 scripts/fetch_aws_price.py ec2 p4de.24xlarge ap-northeast-1
# Bedrock を使う場合のみ。モデルIDはヒアリングで確定したものを指定する
# デフォルトリージョンは ap-northeast-1（東京）
python3 scripts/fetch_aws_price.py bedrock <model_id> ap-northeast-1 --io input
python3 scripts/fetch_aws_price.py bedrock <model_id> ap-northeast-1 --io output
python3 scripts/fetch_aws_price.py s3 standard ap-northeast-1
```

> 補足：`pricing.us-east-1.amazonaws.com` は AWS Price List Bulk API の**配信エンドポイント自体**（全リージョンの料金カタログを `us-east-1` から配信する仕様）。fetch_aws_price.py の内部実装で固定的に使われており、利用リージョンの選択とは無関係なので変更不要。

**重要**：Bedrock 等の生成 AI API を使うかどうかは **ユーザーに確認**する（要否のみ。モデル選定までは聞かない）。研究内容によっては API を一切使わない（GPU 学習・推論のみ）ケースも多く、その場合は API 費用テーブル（行9〜18）を空欄のまま提出する。

API を使う場合のモデルは **`Amazon Bedrock 上の Claude Sonnet 4.6`** を、リージョンは **`ap-northeast-1`（東京）** を**デフォルト**とする（質問項目を増やさないため、モデル・リージョン選定はヒアリングしない）。ユーザーが自発的に具体的なモデル名・他リージョンを出した場合のみ、そちらを優先する。デフォルト適用時は算定根拠の `[備考]` に「研究者は具体モデル未定。本算定では Claude Sonnet 4.6（東京リージョン、JP CRIS対応）を仮置き」と明記する。

> **レガシーモデルの提案禁止**：用途ごとにモデルを分ける場合（例：RAG検索用に軽量モデル、精度検証用に高性能モデル）でも、**Claude 3 系列（Claude 3 Haiku / Claude 3 Sonnet / Claude 3 Opus / Claude 3.5 Haiku / Claude 3.5 Sonnet 等）は提案してはならない**。これらは既にレガシーであり、研究開始時（2026年後半〜2027年）には非推奨またはサポート終了の可能性が高い。用途を分ける場合は現行世代（Haiku 4.5 / Sonnet 4.6 / Opus 4.8）の中で使い分けること。ユーザーが自発的にレガシーモデルを指定した場合のみそちらを使用し、その際も「サポート終了リスクあり」の旨を備考に付記する。

> **JP CRIS と料金の注意**：JP CRIS（日本国内クロスリージョン推論）を利用する場合、追加ルーティングコストはないが、**ソースリージョン（ap-northeast-1）の単価で課金される**。東京リージョンの Bedrock 単価は us-east-1 と異なる場合があるため、必ず `fetch_aws_price.py` で `ap-northeast-1` を指定して東京の正しい単価を取得すること。us-east-1 の単価で代用してはならない。

EC2 GPU インスタンスの典型例：`p4d.24xlarge`（A100 40GB×8）、`p4de.24xlarge`（A100 80GB×8）、`p5.48xlarge`（H100×8）、`p5e.48xlarge`（H200×8）、`g6e.12xlarge`（L40S×4）、`g5.xlarge`（A10G×1）

リージョンは原則 `ap-northeast-1`（東京）。Bedrock も学習・推論用 EC2 も同じく東京を基本とすることで、データ転送料金とレイテンシを最適化する。東京で未提供のサービス・モデルがある場合のみ `us-east-1` 等の他リージョンを検討（その場合は算定根拠の `[備考]` に理由を明記）。オンデマンドが基本。Spot 割引（最大70〜90%OFF）や Savings Plans（1年〜、3年〜）は研究期間6ヶ月には**不向き**（Savings Plans のコミット期間が合わない）。

付帯コスト（S3 Standard 容量・データ転送・EBS gp3・CloudWatch 等）は `fetch_aws_price.py` の該当サービス呼び出しで取得するか、AWS 公式料金ページを参照。

### Step 3. 算定式を組む

```
月額(円) = インスタンス時間単価(USD) × 稼働時間(h/月) × 為替(円/USD) × インスタンス数
総額(円) = 月額 × 月数（※研究期間内、最大6ヶ月）
```

**月数は研究期間（交付決定日〜令和9年2月5日、最大約6ヶ月）を超えないこと。12ヶ月や年額ベースの計算は誤り。**

ヒアリング・メモでは **万円** 表示（`総額 ÷ 10000`）、xlsx書き込み時のみ**千円** 表示（`総額 ÷ 1000`、千円未満切り捨て）。

為替レートは**所属機関の会計基準**に従う（ユーザーに確認）。一般的には1USD=150円や、会計年度ごとの予算用レートを使う。

### Step 3.5. 算定根拠テンプレ

**算定根拠の書き方テンプレ**（xlsx の E列に書き込む文章、**万円単位で表示**）：

```
[用途] LLMファインチューニング学習（学習データ10GB × 3エポック）
[サービス] Amazon EC2 p4de.24xlarge（NVIDIA A100 80GB × 8）
[リージョン] ap-northeast-1（東京）、オンデマンド
[単価] $40.96/時（2026年4月時点、AWS Price List Bulk API より `scripts/fetch_aws_price.py ec2 p4de.24xlarge ap-northeast-1` で取得）
[稼働時間] 80時間/月 × 6ヶ月 = 480時間
[再実行率] 20% バッファ
[計算] $40.96/時 × 480時間 × 1.20 × 150円/USD = 3,538,944円 ≒ 353.9 万円
[備考] 複数回のハイパーパラメータ探索を含む
```

（xlsx 4枚目「金額(千円)」欄には `3,538` と入れる）

単価の情報源は `scripts/fetch_aws_price.py`（AWS Price List Bulk API）で都度取得し、取得日を `[単価]` 行に明記する。取得できない SKU は AWS 公式料金ページ（https://aws.amazon.com/ec2/pricing/ 等）の参照日を明記。**Pricing Calculator の共有URLは不要**。

**選定理由の書き方例（C列）**：

```
70Bパラメータ級LLMのフルファインチューニング（学習データ10GB）には、
80GB VRAM×8 GPUが必要。A100 80GB搭載の p4de.24xlarge を選定。
H100 (p5) も検討したが、本研究では FP16 学習で充分であり、
価格優位な A100 で要件を満たすため。
```

### Step 4. API費用の特殊ケース（Bedrock 等）

Bedrock を使う場合は「API費用」として4枚目上段（行9〜18）に記載。**Bedrock を使うかどうか／どのモデルを使うかは必ずユーザーにヒアリング**してから記入する：

- 処理対象（D列）：**AWSサービス名**（デフォルト：`Amazon Bedrock (Claude Sonnet 4.6)`、ユーザーが他モデルを指定した場合はそちら）。具体的な用途・処理対象は算定根拠（F列）の `[備考]` に書く
- 金額（E列、千円）：月あたりトークン数 × 1Mトークン単価 × 月数 × 為替（×10で千円換算）
- 算定根拠（F列、merged F:G）：下記テンプレに沿って記述。審査員が一目で何のAPIか分かるよう **API種類を日本語**で、さらに**リージョン・単価・稼働時間（またはトークン量）**を必ず含める

```
[API種類] 大規模言語モデルAPI（文献要約・情報抽出用）
[サービス] Amazon Bedrock (Claude Sonnet 4.6)  ← デフォルト。ヒアリングで他モデル指定があればそれを書く
[リージョン] ap-northeast-1（東京）  ← デフォルト。東京未提供等の理由がある場合のみ他リージョン
[単価] 入力 $X.XX/1M tokens、出力 $YY.YY/1M tokens（取得日, AWS Price List Bulk API取得）
[想定量] 入力 N tok/月、出力 M tok/月 × Kヶ月
[計算] (N×単価 + M×単価)/1M × Kヶ月 × 為替 × 1.2(再実行) = …円 ≒ … 万円
[備考] 研究者は具体モデル未定。本算定では Claude Sonnet 4.6（東京リージョン、JP CRIS対応。CRIS利用時もソースリージョン=東京の単価で課金）を仮置きし、応募後にモデル選定を詰める想定。（研究での具体的な利用シーン）
```

時間ベースの API（推論エンドポイント等）の場合は `[稼働時間] X時間/月 × Yヶ月` の行も追加する。

**データ取得・照会・連携のための API**（例：PubMed API, GenBank API の有償プラン）も同じ書式で記載。AWSサービス以外の場合はサービス提供元名を先頭に書く（例：`[サービス] NCBI E-utilities API (有償プラン)`）。

**データ取得・照会・連携のための API**（例：PubMed API, GenBank API の有償プラン）も同じ書式で記載。

### Step 5. 合計と3枚目との整合

- 4枚目の API費用合計（行9〜18 E列）＋計算資源費用合計（行22〜31 F列）≦ 3枚目「その他」総計（`E89`）
- 3枚目「その他」欄（行69〜88）の「事項」には「AWS計算資源費用（詳細は4枚目）」「Bedrock API利用料（詳細は4枚目）」等と記載
- 設備備品費として計上した計算資源（GPUカード等）は4枚目「計算資源費用」テーブルにも記載するが、3枚目「その他」総計には**含めない**（= 3枚目では設備備品費に計上）
- AWS 以外の「その他」費目（英文校正、論文掲載料、HPCI 特定研究有償課題 等）も3枚目「その他」に含むので、`4枚目合計 < 3枚目その他総計` となるケースもある
- `validate.py` がこの整合性を自動で確認する

## 注意

- 物理的に GPU サーバ・ワークステーションを購入する場合は**設備備品費（3枚目）** に計上し、その**詳細は4枚目「計算資源費用」テーブルにも書く**（第2回公募の注意事項で明記された）。3枚目その他総計と4枚目合計は二重計上とならないように分けて整合させる。
- **為替変動リスク**は選定理由か算定根拠に一言触れておくと親切（「為替150円/USDで算定。変動時は費目間流用で吸収」等）。
- **xlsx 様式**の金額欄は**千円単位・千円未満切り捨て**（様式固定）。**ヒアリングや内部メモは万円単位**で扱い、書き込む時だけ `×10` で千円に換算する。
